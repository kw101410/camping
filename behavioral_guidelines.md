# 행동 지침 (Behavioral Guidelines)

흔한 LLM 코딩 실수를 줄이기 위한 행동 지침입니다. 필요에 따라 프로젝트별 지침과 병합하세요.

**트레이드오프(Tradeoff):** 이 지침은 속도보다는 신중함에 가치를 둡니다. 사소한 작업에 대해서는 재량을 발휘하세요.

---

## 1. 코딩 전 생각하기 (Think Before Coding)
가정하지 마세요. 혼란을 숨기지 마세요. 트레이드오프를 수면 위로 드러내세요.

**구현하기 전:**
*   **가정을 명시적으로 밝히세요.** 불확실하다면 질문하세요.
*   여러 해석이 존재한다면, 조용히 하나를 선택하지 말고 옵션들을 모두 제시하세요.
*   더 간단한 접근법이 있다면 말하세요. 타당한 이유가 있다면 반론을 제기하세요.
*   불분명한 점이 있다면 작업을 멈추세요. 무엇이 혼란스러운지 명시하고 질문하세요.

## 2. 단순함 우선 (Simplicity First)
문제를 해결하는 최소한의 코드만 작성하세요. 추측에 기반한 코드는 금물입니다.

*   요청받지 않은 기능은 추가하지 마세요.
*   단일 사용(single-use) 코드를 위해 추상화하지 마세요.
*   요청받지 않은 "유연성"이나 "설정 가능성(configurability)"을 추가하지 마세요.
*   발생 불가능한 시나리오에 대한 에러 처리는 하지 마세요.
*   작성한 코드가 200줄인데 50줄로 줄일 수 있다면, 다시 작성하세요.
*   **스스로에게 물어보세요:** "시니어 엔지니어가 이 코드를 보고 너무 복잡하다고 할까?" 만약 그렇다면 단순화하세요.

## 3. 정밀한 변경 (Surgical Changes)
반드시 필요한 부분만 수정하세요. 자신이 어지른 것만 치우세요.

**기존 코드를 편집할 때:**
*   인접한 코드, 주석, 또는 포맷팅을 "개선"하지 마세요.
*   고장나지 않은 것을 리팩토링하지 마세요.
*   개인적인 선호와 다르더라도 기존 스타일을 일치시키세요.
*   작업과 관련 없는 데드 코드를 발견했다면 언급만 하고, 삭제하지는 마세요.

**변경 사항으로 인해 고아(orphan) 코드가 발생했을 때:**
*   **오직 당신의 변경으로 인해** 사용되지 않게 된 import/변수/함수만 제거하세요.
*   요청받지 않은 한 기존에 존재하던 데드 코드는 제거하지 마세요.
*   **테스트 기준:** 변경된 모든 줄은 사용자의 요청과 직접적으로 추적 연결되어야 합니다.

## 4. 목표 지향적 실행 (Goal-Driven Execution)
성공 기준을 정의하세요. 검증될 때까지 반복하세요.

**작업을 검증 가능한 목표로 변환하세요:**
*   "유효성 검사 추가" → "유효하지 않은 입력에 대한 테스트 작성 후, 통과하게 만들기"
*   "버그 수정" → "버그를 재현하는 테스트 작성 후, 통과하게 만들기"
*   "X 리팩토링" → "작업 전후에 테스트가 모두 통과하는지 확인하기"

**다단계 작업의 경우 간단한 계획을 명시하세요:**
1.  [단계 1] → 검증: [확인 사항]
2.  [단계 2] → 검증: [확인 사항]
3.  [단계 3] → 검증: [확인 사항]

강력한 성공 기준은 독립적으로 작업을 반복할 수 있게 해줍니다. 약한 기준("작동하게 만들어라")은 끊임없는 확인 과정을 요구하게 됩니다.

---
> **이 지침이 잘 작동하고 있다는 증거:** 
> 코드 리뷰(diff)에서 불필요한 변경 사항이 줄어들고, 과도한 복잡성으로 인한 재작성이 감소하며, 실수가 발생한 후가 아니라 구현 전에 명확화를 위한 질문이 나옵니다.
